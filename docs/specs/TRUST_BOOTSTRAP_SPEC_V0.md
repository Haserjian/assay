# Trust Bootstrap Spec v0

Version: 0.2.0
Created: 2026-03-28
Status: Design draft — not yet implemented

---

## Problem

The Assay trust evaluation surface works: signers.yaml, acceptance.yaml,
and enforce-trust are wired end-to-end in assay-verify-action. But a
first-time user must reverse-engineer the YAML shapes from action source
code. There is no 5-minute path from "I want CI trust" to "my workflow
enforces signer policy."

The bootstrap must eliminate "what shape does this policy even have?"
for every profile, not just the advanced ones.

## Command Shape

```bash
assay trust bootstrap --profile <minimal|reviewer|strict> [options]
```

**Ownership split:**
- `assay-toolkit` CLI owns the `trust bootstrap` command (generates files locally)
- `assay-verify-action` owns canonical example policy files (shipped in `examples/trust/`)
- The command does not depend on the action repo at runtime — it embeds templates

**Options:**
| Flag | Default | Purpose |
|------|---------|---------|
| `--profile` | `minimal` | Template profile |
| `--output-dir` | `.` | Where to write files |
| `--trust-target` | `ci_gate` | Target for acceptance rules |
| `--signer-id` | `assay-ci-signer` | Placeholder signer identity |
| `--dry-run` | false | Show what would be written |

## Emitted Files (All Profiles)

Every profile emits the same four files. The profiles differ in
posture (permissive → advisory → enforced) and in guidance comments.

```
<output-dir>/
├── .github/workflows/assay-verify.yml
├── trust/
│   ├── signers.yaml
│   ├── acceptance.yaml
│   └── README.md
```

## Design Invariant

All profiles emit trust policy files. A user who runs any variant
of `assay trust bootstrap` will never need to reverse-engineer YAML
shape. The shape is always visible, always commented, and always
functional out of the box.

## Profile Matrix

| Aspect | `minimal` | `reviewer` | `strict` |
|--------|-----------|------------|----------|
| Trust files emitted | Yes | Yes | Yes |
| Trust evaluation in workflow | Yes (advisory) | Yes (advisory) | Yes (enforced) |
| `enforce-trust` in workflow | `false` | `false` | `true` |
| Unrecognized signer | `accept` | `warn` | `reject` |
| Revoked signer | `warn` | `reject` | `reject` |
| Gallery 05/06 references | In README only | In README + inline YAML comments | In README |
| Guidance tone | "Get started" | "Build toward reviewer packets" | "Lock down" |

### `minimal` — trust-aware from the start, permissive posture

**Audience:** Teams trying Assay for the first time. Want to see trust
evaluation working without any risk of blocking their pipeline.

**Workflow:**
```yaml
- uses: Haserjian/assay-verify-action@v1
  with:
    pack-path: "proof_pack_*/"
    require-pack: true
    require-claim-pass: true
    comment-on-pr: true
    trust-target: ci_gate
    trust-policy-dir: trust/
```

**trust/signers.yaml:**
```yaml
# Assay Trust Policy — Signer Registry
# Generated by: assay trust bootstrap --profile minimal
#
# This file defines who is allowed to sign proof packs.
#
# To register your signer:
#   1. Run: assay key list
#   2. Find the SHA-256 fingerprint of your active key
#   3. Uncomment the example entry below and replace the placeholder
#   4. Commit and push — your next PR will show "authorized" in the trust summary

version: 1

signer_classes:
  ci-org:
    description: "Organization CI signer"
    allowed_targets: [ci_gate, publication]
  local-dev:
    description: "Developer local key (not sufficient for CI gate)"
    allowed_targets: [local_verify]
    not_sufficient_for: [ci_gate, publication]

signers:
  # Uncomment and replace with your actual fingerprint:
  # - signer_id: "my-ci-signer"
  #   signer_class: "ci-org"
  #   fingerprint: "replace-with-output-of-assay-key-list"
  #   lifecycle: "active"
  #   grants:
  #     - artifact_class: "proof_pack"
  #       purpose: "*"
```

**trust/acceptance.yaml:**
```yaml
# Assay Trust Policy — Acceptance Rules
# Generated by: assay trust bootstrap --profile minimal
#
# Rules are evaluated top-to-bottom, first match wins.
#
# Minimal profile: permissive posture.
# All signers accepted. Trust evaluation runs and reports
# in the PR comment, but nothing is blocked.
# Upgrade to --profile reviewer or --profile strict when ready.

rules:
  # Accept authorized signers
  - artifact_class: "*"
    verification_level: "signature_verified"
    authorization_status: "authorized"
    target: "ci_gate"
    decision: "accept"
    reason: "Signed by authorized CI signer"

  # Accept unrecognized signers (permissive — no signer registered yet)
  - artifact_class: "*"
    verification_level: "*"
    authorization_status: "unrecognized"
    target: "ci_gate"
    decision: "accept"
    reason: "Signer not yet registered — add fingerprint to trust/signers.yaml"

  # Warn on revoked signers (even permissive mode flags revocations)
  - artifact_class: "*"
    verification_level: "*"
    authorization_status: "revoked"
    target: "*"
    decision: "warn"
    reason: "Signer has been revoked"
```

**First-run behavior:** CI passes. PR comment shows trust summary with
"Signer not yet registered" as the acceptance reason. The message is
guidance, not failure. The user sees the trust surface working without
any surprise breakage.

**What the user learns:** The shape of trust policy, how trust
evaluation appears in PR comments, and what their next step is (register
a signer). Zero reverse-engineering.

### `reviewer` — advisory posture, reviewer packet orientation

**Audience:** Teams building toward reviewer packets or compliance
evidence. Want trust evaluation to surface warnings when signers are
unknown.

**Workflow:** Same as minimal, identical flags. The difference is
entirely in the acceptance rules.

```yaml
- uses: Haserjian/assay-verify-action@v1
  with:
    pack-path: "proof_pack_*/"
    require-pack: true
    require-claim-pass: true
    comment-on-pr: true
    trust-target: ci_gate
    trust-policy-dir: trust/
```

**trust/signers.yaml:**
```yaml
# Assay Trust Policy — Signer Registry
# Generated by: assay trust bootstrap --profile reviewer
#
# This file defines who is allowed to sign proof packs.
# The reviewer profile expects you to register your CI signer
# before reviewer packets will show clean trust status.
#
# To register your signer:
#   1. Run: assay key list
#   2. Find the SHA-256 fingerprint of your active key
#   3. Uncomment the example entry below and replace the placeholder
#   4. Commit and push
#
# For reviewer packet examples, see:
#   Scenario 05 (buyer-facing):     https://github.com/Haserjian/assay-proof-gallery/tree/main/gallery/05-reviewer-packet-gaps
#   Scenario 06 (NAIC AISET):       https://github.com/Haserjian/assay-proof-gallery/tree/main/gallery/06-naic-aiset-mapping

version: 1

signer_classes:
  ci-org:
    description: "Organization CI signer bound to GitHub Actions"
    allowed_targets: [ci_gate, publication]
  local-dev:
    description: "Developer local key (not sufficient for CI gate or reviewer packets)"
    allowed_targets: [local_verify]
    not_sufficient_for: [ci_gate, publication]

signers:
  # Uncomment and replace with your actual fingerprint:
  # - signer_id: "my-ci-signer"
  #   signer_class: "ci-org"
  #   fingerprint: "replace-with-output-of-assay-key-list"
  #   lifecycle: "active"
  #   grants:
  #     - artifact_class: "proof_pack"
  #       purpose: "*"
  #   notes: "Primary CI signer for reviewer packet evidence"
```

**trust/acceptance.yaml:**
```yaml
# Assay Trust Policy — Acceptance Rules
# Generated by: assay trust bootstrap --profile reviewer
#
# Rules are evaluated top-to-bottom, first match wins.
#
# Reviewer profile: advisory posture.
# Trust evaluation runs and warns on unregistered signers.
# Pipeline does not fail — warnings appear in PR comments.
#
# This posture is appropriate while building toward reviewer
# packets. Upgrade to --profile strict when signer registration
# is complete and you want enforcement.
#
# Reviewer packet examples:
#   https://github.com/Haserjian/assay-proof-gallery (scenarios 05, 06)

rules:
  # Accept authorized signers
  - artifact_class: "proof_pack"
    verification_level: "signature_verified"
    authorization_status: "authorized"
    target: "ci_gate"
    decision: "accept"
    reason: "Signed by authorized CI signer"

  # Warn on recognized but not-yet-authorized signers
  - artifact_class: "proof_pack"
    verification_level: "signature_verified"
    authorization_status: "recognized"
    target: "ci_gate"
    decision: "warn"
    reason: "Signer recognized but not authorized for ci_gate — check grants in signers.yaml"

  # Warn on unrecognized signers
  - artifact_class: "*"
    verification_level: "*"
    authorization_status: "unrecognized"
    target: "ci_gate"
    decision: "warn"
    reason: "Unknown signer — register fingerprint in trust/signers.yaml"

  # Reject revoked signers
  - artifact_class: "*"
    verification_level: "*"
    authorization_status: "revoked"
    target: "*"
    decision: "reject"
    reason: "Signer has been revoked"
```

**First-run behavior:** CI passes. PR comment shows trust summary with
a **warning**: "Unknown signer — register fingerprint in trust/signers.yaml".
The warning is visible and actionable but does not block the pipeline.

**How reviewer differs from minimal (what the user feels):**
- Minimal: everything accepts quietly. Trust summary says "accepted"
  even with no signer. Low friction, low signal.
- Reviewer: unrecognized signers trigger visible warnings. The PR
  comment now has amber/yellow status that says "you need to register."
  The user is prompted to act without being blocked.

This is the teaching-by-nudge model: visible signal, no breakage.

### `strict` — enforced posture, fail-closed on unknown signers

**Audience:** Teams with signer infrastructure in place. Want the
pipeline to fail if a proof pack is signed by an unregistered key.

**Workflow:**
```yaml
- uses: Haserjian/assay-verify-action@v1
  with:
    pack-path: "proof_pack_*/"
    require-pack: true
    require-claim-pass: true
    comment-on-pr: true
    trust-target: ci_gate
    trust-policy-dir: trust/
    enforce-trust: true
```

**trust/signers.yaml:** Same structure as reviewer (with gallery links
and commented example), but the guidance comment says:

```yaml
# Assay Trust Policy — Signer Registry
# Generated by: assay trust bootstrap --profile strict
#
# IMPORTANT: The strict profile enforces trust. Proof packs signed
# by unregistered signers will FAIL the CI pipeline.
#
# You MUST register at least one signer before the first PR,
# or the pipeline will reject all proof packs.
#
# To register your signer:
#   1. Run: assay key list
#   2. Find the SHA-256 fingerprint of your active key
#   3. Uncomment the example entry below and replace the placeholder
#   4. Commit trust/signers.yaml BEFORE pushing proof packs
```

(Remainder of file identical to reviewer.)

**trust/acceptance.yaml:**
```yaml
# Assay Trust Policy — Acceptance Rules
# Generated by: assay trust bootstrap --profile strict
#
# Rules are evaluated top-to-bottom, first match wins.
#
# Strict profile: enforced posture.
# Unregistered signers are REJECTED. The CI pipeline will fail
# if a proof pack is signed by an unknown key.
#
# Make sure at least one signer is registered in signers.yaml
# before pushing proof packs.

rules:
  # Accept authorized signers
  - artifact_class: "proof_pack"
    verification_level: "signature_verified"
    authorization_status: "authorized"
    target: "ci_gate"
    decision: "accept"
    reason: "Signed by authorized CI signer"

  # Warn on recognized but not-yet-authorized signers
  - artifact_class: "proof_pack"
    verification_level: "signature_verified"
    authorization_status: "recognized"
    target: "ci_gate"
    decision: "warn"
    reason: "Signer recognized but not authorized for ci_gate — check grants"

  # REJECT unrecognized signers (strict enforcement)
  - artifact_class: "*"
    verification_level: "*"
    authorization_status: "unrecognized"
    target: "ci_gate"
    decision: "reject"
    reason: "Unknown signer — register fingerprint in trust/signers.yaml before pushing"

  # Reject revoked signers
  - artifact_class: "*"
    verification_level: "*"
    authorization_status: "revoked"
    target: "*"
    decision: "reject"
    reason: "Signer has been revoked"
```

**First-run behavior:** If no signer registered, CI **fails** with a
clear rejection message. This is intentional — strict means strict.
The signers.yaml comment warns about this upfront.

## First-Run UX

All profiles print the same structure, varying in posture description.

**`assay trust bootstrap --profile reviewer`** (example):

```
Assay Trust Bootstrap (reviewer profile)

Writing files:
  trust/signers.yaml              signer registry (placeholder — register your key)
  trust/acceptance.yaml           acceptance rules (advisory — warns on unknown signers)
  trust/README.md                 setup guide + gallery links
  .github/workflows/assay-verify.yml   CI workflow with trust evaluation

Trust posture: advisory
  - Authorized signers: accepted
  - Unknown signers: warned (visible in PR, pipeline passes)
  - Revoked signers: rejected

Next steps:
  1. Run: assay key list
  2. Copy your fingerprint into trust/signers.yaml (replace the placeholder)
  3. Commit and push
  4. Open a PR with a proof pack — trust evaluation appears in the PR comment

Reviewer packet examples:
  https://github.com/Haserjian/assay-proof-gallery
  Scenario 05: reviewer-packet-gaps (buyer-facing)
  Scenario 06: naic-aiset-mapping (compliance/regulatory)
```

**`assay trust bootstrap --profile minimal`** — same structure, but:
```
Trust posture: permissive
  - All signers: accepted (trust summary shown, nothing blocked)
  - Revoked signers: warned
```

**`assay trust bootstrap --profile strict`** — same structure, but:
```
Trust posture: enforced
  - Authorized signers: accepted
  - Unknown signers: REJECTED (pipeline fails)
  - Revoked signers: rejected

  ⚠ Register at least one signer BEFORE pushing proof packs,
    or the pipeline will reject all packs.
```

## trust/README.md (All Profiles)

```markdown
# Trust Policy

This directory contains the signer registry and acceptance rules
for Assay proof pack verification in CI.

## Quick start

1. Find your signer fingerprint:
   ```
   assay key list
   ```

2. Add the fingerprint to `signers.yaml` under `signers:`
   (replace the commented placeholder)

3. Commit and push. Trust evaluation appears in the next PR comment.

## Files

| File | Purpose |
|------|---------|
| `signers.yaml` | Who is allowed to sign proof packs |
| `acceptance.yaml` | What decisions apply per signer status and target |

## How trust evaluation works

1. `assay verify-pack` checks pack integrity and signer identity
2. The signer fingerprint is looked up in `signers.yaml`
3. The acceptance rules in `acceptance.yaml` determine the decision
4. The decision appears in the PR comment (accept / warn / reject)

## Verification levels

| Level | Meaning |
|-------|---------|
| `signature_verified` | Ed25519 signature valid, signer fingerprint extracted |
| `hash_verified` | File hashes match manifest, no signature check |
| `unverified` | No verification performed |

## Reviewer packet examples

- [Scenario 05: reviewer-packet-gaps](https://github.com/Haserjian/assay-proof-gallery/tree/main/gallery/05-reviewer-packet-gaps) — buyer-facing
- [Scenario 06: naic-aiset-mapping](https://github.com/Haserjian/assay-proof-gallery/tree/main/gallery/06-naic-aiset-mapping) — compliance/regulatory

## Upgrading trust posture

| From | To | Change |
|------|----|--------|
| minimal → reviewer | Edit `acceptance.yaml`: change `unrecognized` decision from `accept` to `warn` |
| reviewer → strict | Edit workflow: add `enforce-trust: true`. Edit `acceptance.yaml`: change `unrecognized` decision from `warn` to `reject` |
```

## Connection to Proof Gallery 05/06

The reviewer profile's signers.yaml and trust/README.md link to gallery
scenarios 05 and 06 as canonical destinations. The bootstrap teaches
policy setup; the gallery teaches what a finished reviewer packet looks
like.

Adoption flow:
1. `assay trust bootstrap --profile reviewer` → trust policy + workflow
2. User registers signer, pushes proof pack → CI shows trust evaluation
3. User inspects gallery 05 → sees reviewer packet shape
4. User builds their own reviewer packet via `assay reviewer`
5. Trust evaluation in CI validates the nested proof pack signer

## Design Questions — Answered

**Should bootstrap live in assay-toolkit CLI or assay-verify-action?**
assay-toolkit CLI. The command generates local files. The action repo
owns the example/reference versions (in `examples/trust/`) for
documentation.

**Should example trust-policy files ship in assay-verify-action?**
Yes. Add `examples/trust/` with signers.yaml and acceptance.yaml so
the action README can reference concrete shapes inline.

**What is the minimum policy shape for a first-time user?**
The minimal profile. It emits fully-shaped trust files with a
commented placeholder signer. First CI run accepts everything and
shows the trust summary. No reverse-engineering. No breakage.

**How does reviewer differ from minimal without exploding complexity?**
One rule change: unrecognized signers go from `accept` to `warn`.
The user sees an amber warning in their PR comment instead of a
quiet accept. That is the nudge that teaches signer registration.
Same file structure, same shapes. The posture is the only variable.

**Should the first CI run warn or pass-with-guidance?**
Depends on profile:
- Minimal: pass-with-guidance (accept + reason message in PR comment)
- Reviewer: pass with visible warning (warn + reason message)
- Strict: fail with clear rejection (reject + reason message)

No profile produces a silent pass with no trust information. Every
profile shows the trust summary. The difference is severity.

## Out of Scope (v0)

- Key generation or rotation (`assay key generate` already exists)
- OIDC/Sigstore integration
- Multi-repo policy federation
- Policy linting or schema validation
- Auto-detection of existing trust config
- Lockfile generation (`assay lock write` already exists, separate concern)
- Comparability contract setup (different adoption surface)

## Risks and Unknowns

1. **Placeholder signer may be committed as-is.** Users might push
   the commented example without replacing the fingerprint. Mitigation:
   the placeholder value `replace-with-output-of-assay-key-list` will
   never match a real fingerprint, so it produces the correct
   "unrecognized" behavior rather than a false positive.

2. **Workflow template drift.** The embedded workflow template must be
   updated when assay-verify-action adds inputs. Mitigation: pin the
   action version (e.g. `@v1`) in the template.

3. **Profile naming.** "reviewer" implies reviewer packets specifically,
   but the trust policy applies to all proof packs. Alternative names:
   "advisory" / "enforced". Current names chosen for adoption-path
   clarity over taxonomic precision.

## Smallest Shippable Implementation Plan

1. **Add example trust files to assay-verify-action** (~30 min)
   - `examples/trust/signers.yaml` (reviewer profile version)
   - `examples/trust/acceptance.yaml` (reviewer profile version)
   - Update action README to reference them with inline examples

2. **Implement `assay trust bootstrap` in assay-toolkit** (~2 hours)
   - Add `trust_bootstrap_cmd` to commands.py
   - Embed templates for all three profiles (signers, acceptance, README, workflow)
   - Write files with profile-specific comments and posture
   - Print first-run UX summary with posture description and next steps
   - Refuse to overwrite existing trust/ directory without `--force`

3. **Test locally** (~30 min)
   - Run each profile, inspect emitted files
   - Verify minimal produces accept-with-guidance on unregistered signer
   - Verify reviewer produces warn on unregistered signer
   - Verify strict produces reject on unregistered signer
   - Verify placeholder fingerprint never accidentally matches
